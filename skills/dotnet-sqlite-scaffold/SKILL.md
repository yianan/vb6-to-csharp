# ASP.NET Core 10 + EF Core + SQLite + cookie auth — scaffold recipe

This is the exact recipe used to build the seed library app's backend. It's known-good as of 2026-05.

## Prerequisites

- **.NET 10 SDK**: `brew install dotnet` (macOS) — confirm `dotnet --version` ≥ 10.0.x. Avoid .NET 9 (EOL May 2026).
- **dotnet-ef** matching SDK major: `dotnet tool install --global dotnet-ef --version "10.*"`. Add `~/.dotnet/tools` to PATH.

### WSL2 notes

For WSL2-runnable migrations, keep the target stack Linux-safe:

- Good fits: ASP.NET Core, EF Core, SQLite/PostgreSQL, React/Vite, console/worker services.
- Avoid runtime dependencies on WinForms, WPF, VB6 runtime, COM registration, OCX controls, Windows registry APIs, and Access OLEDB/ACE providers.
- Run `dotnet --info`, `node --version`, and `npm --version` before scaffolding.
- If system `dotnet` is broken or mixed across package sources, use a local SDK and call it by absolute path, for example `./.dotnet/dotnet build`.
- If WSL has old Node or Node without `npm`, use a local Node runtime and prefix frontend commands with `PATH="$PWD/.node/bin:$PATH"`.
- Installs under `/mnt/c` can be slow. Prefer the native Linux filesystem for large frontend dependency trees when practical.

### macOS Tahoe (26.x) workaround

The `dotnet-ef` apphost shim refuses to start on macOS 26 because the RID lookup fails:

```
missing_runtime=true&os=osx.26&apphost_version=10.0.7
```

The runtime *is* installed; the shim just can't find it. Fix:

```sh
export DOTNET_ROOT=/opt/homebrew/opt/dotnet/libexec
```

Bake into `~/.zshrc` or a project `.envrc`. Required for the `dotnet-ef` global tool only — `dotnet build` / `dotnet run` work without it.

## Scaffold

```sh
mkdir -p backend && cd backend
dotnet new webapi --use-minimal-apis -n Library.Api -o Library.Api
cd Library.Api
dotnet add package Microsoft.EntityFrameworkCore.Sqlite --version "10.*"
dotnet add package Microsoft.EntityFrameworkCore.Design --version "10.*"
dotnet add package BCrypt.Net-Next
```

`Microsoft.AspNetCore.Authentication.Cookies` is in the framework — no package needed.

## `Program.cs` template

```csharp
using System.Text.Json.Serialization;
using Library.Api.Data;
using Library.Api.Endpoints;
using Library.Api.Services;
using Microsoft.AspNetCore.Authentication.Cookies;
using Microsoft.AspNetCore.Http.Json;
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.Configure<JsonOptions>(o =>
{
    o.SerializerOptions.ReferenceHandler = ReferenceHandler.IgnoreCycles;
    o.SerializerOptions.DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull;
});

builder.Services.AddDbContext<LibraryDbContext>(opt =>
    opt.UseSqlite(builder.Configuration.GetConnectionString("Sqlite") ?? "Data Source=library.db"));

builder.Services.AddScoped<LoanService>();
builder.Services.AddScoped<FineCalculator>();

builder.Services
    .AddAuthentication(CookieAuthenticationDefaults.AuthenticationScheme)
    .AddCookie(o =>
    {
        o.Cookie.Name = "library.auth";
        o.Cookie.HttpOnly = true;
        o.Cookie.SameSite = SameSiteMode.Lax;
        o.ExpireTimeSpan = TimeSpan.FromDays(7);
        o.SlidingExpiration = true;
        o.Events.OnRedirectToLogin = ctx => { ctx.Response.StatusCode = 401; return Task.CompletedTask; };
        o.Events.OnRedirectToAccessDenied = ctx => { ctx.Response.StatusCode = 403; return Task.CompletedTask; };
    });
builder.Services.AddAuthorization();

builder.Services.AddCors(o => o.AddDefaultPolicy(p => p
    .WithOrigins("http://localhost:5180")  // adjust for your Vite port
    .AllowAnyHeader()
    .AllowAnyMethod()
    .AllowCredentials()));

var app = builder.Build();

using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<LibraryDbContext>();
    db.Database.Migrate();
    Seed.Run(db);
}

app.UseCors();
app.UseAuthentication();
app.UseAuthorization();

app.MapAuthEndpoints();
app.MapBookEndpoints();
// … other endpoint groups

app.MapGet("/api/health", () => Results.Ok(new { ok = true }));

app.Run();

public partial class Program { }  // for WebApplicationFactory in tests
```

## DbContext template

```csharp
public class LibraryDbContext(DbContextOptions<LibraryDbContext> options) : DbContext(options)
{
    public DbSet<User> Users => Set<User>();
    public DbSet<Book> Books => Set<Book>();
    // …

    protected override void OnModelCreating(ModelBuilder b)
    {
        b.Entity<User>().HasIndex(u => u.Username).IsUnique();
        b.Entity<Book>().Property(x => x.Cost).HasPrecision(10, 2);
        // FKs, cascade rules, indexes…
    }
}
```

## Auth endpoint template

```csharp
g.MapPost("/login", async (LoginDto dto, LibraryDbContext db, HttpContext http) =>
{
    var user = await db.Users.FirstOrDefaultAsync(u => u.Username == dto.Username);
    if (user is null || !BCrypt.Net.BCrypt.Verify(dto.Password, user.PasswordHash)) return Results.Unauthorized();

    var claims = new[]
    {
        new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
        new Claim(ClaimTypes.Name, user.Username),
        new Claim(ClaimTypes.Role, user.Role)
    };
    var identity = new ClaimsIdentity(claims, CookieAuthenticationDefaults.AuthenticationScheme);
    await http.SignInAsync(CookieAuthenticationDefaults.AuthenticationScheme, new ClaimsPrincipal(identity));
    return Results.Ok(new { user.Id, user.Username, user.Role });
});

// Protect endpoint groups with .RequireAuthorization():
var g = app.MapGroup("/api/books").RequireAuthorization();
```

## Migration commands

```sh
export DOTNET_ROOT=/opt/homebrew/opt/dotnet/libexec   # macOS 26 workaround
dotnet-ef migrations add Initial
dotnet-ef database update
```

`Program.cs` calls `db.Database.Migrate()` at startup so the deploy story is "ship the binary" — no separate migration step.

## Why these choices

- **EF Core SQLite** — zero-deploy database, perfect for local-first apps. Move to Postgres later via a connection-string change if you need concurrency at scale.
- **Cookie auth, not JWT** — same-origin via Vite proxy means cookies are simpler, more secure (HttpOnly), and stateless tokens have no story for revocation.
- **Minimal APIs, not controllers** — less ceremony for small CRUD APIs. Controllers if you need per-action `[Authorize]` filters or fine-grained model binding.
- **`db.Database.Migrate()` at startup** — fine for single-instance local apps; for multi-instance production, run migrations separately.
- **`ReferenceHandler.IgnoreCycles`** — cheap fix for entity serialization. DTOs are the proper long-term answer.
